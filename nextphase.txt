This is a good point to freeze scope. If you don't, the project will slowly become a generic tournament platform and never finish.

I'd recommend defining **V1** as:

```text
One Season
One Apartment
2-4 Categories
Badminton Only
Singles + Doubles
One Active Tournament Per Category
One Shared Admin
Unlimited Viewers
```

Here is a requirements document you can save as `req.md`.

# Badminton Tournament Manager - Requirements & Roadmap

## 1. Objective

Build a lightweight web application to manage apartment badminton tournaments.

The application should allow tournament organizers to create tournaments, manage players, enter scores, generate standings, run knockout rounds, and publish results.

The application will be hosted on GitHub Pages and use Supabase for persistence and synchronization.

---

# 2. Scope (V1)

## Included

### Tournament Types

* Badminton Singles
* Badminton Doubles

### Categories

Examples:

* Junior Singles
* Junior Doubles
* Senior Boys Singles
* Senior Girls Singles
* Senior Doubles

Maximum 2-4 active categories.

### Users

#### Admin

Can:

* Create tournament
* Add/edit players
* Generate groups
* Enter scores
* Upload champion photos
* Reset category
* Import/export tournament

#### Viewer

Can:

* View groups
* View fixtures
* View standings
* View knockout bracket
* View results

Cannot:

* Edit scores
* Reset tournament
* Start tournament
* Upload photos
* Modify categories

# 3. Tournament Flow

## Phase 1 - Setup

Admin:

* Select category
* Enter player/team count
* Enter player/team names

Validation:

* No duplicate names
* Minimum 2 entries
* Maximum 20 entries

Output:

* Tournament created

## Phase 2 - Group Allocation

System generates groups.

Rules:

### 2-5 Players

* Single group

### 6-10 Players

* 2 groups

### 11-20 Players

* 4 groups

Output:

* Group assignment

* P2 vs P3

# 5. Security

## Viewer

Must not be able to:

* Start tournament
* Reset tournament
* Change groups
* Enter scores
* Import/export
* Upload photos

UI protection is not sufficient.

Every mutating function must validate:

```javascript
if (!_isAdmin) return;
```

## Admin

Authenticated through Supabase Auth.

One architectural recommendation before you write more code:

**Do not add other sports until Phase 2 is complete.**

Right now your biggest risk is not missing features. It is:

```text
Admin/Viewer security
State synchronization
Category switching
Realtime updates
```

Those four areas should be rock-solid before adding Table Tennis, Chess, Cricket, etc. Otherwise you'll multiply the bugs across every sport.
