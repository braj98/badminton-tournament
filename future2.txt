# Requirement - Published Event Report

## Objective

Create a professional, shareable Event Report that celebrates the successful completion of an event.

Unlike tournament data, the Event Report is a **published document** generated from tournament state, reviewed by an administrator, and then made available to all viewers.

The Event Report represents the official record of the event.

---

# Design Philosophy

Separate live operational data from published event content.

## Live Data

Tournament State

Purpose:

* Tournament management
* Score updates
* Match lifecycle
* Bracket generation
* Champion determination

This remains editable throughout the event.

---

## Published Data

Event Report

Purpose:

* Celebrate the event
* Share with participants
* Preserve memories
* Provide statistics
* Appreciate organizers and volunteers

Once published, viewers always see the published version.

---

# Administrator Actions

## Generate Report

Generate the latest report using current tournament data.

This operation does not publish the report.

Purpose:

Preview and verification.

---

## Preview Report

Visible only to administrators.

Allows review before publishing.

Administrator can:

* Verify champions
* Verify statistics
* Upload photos
* Edit appreciation message

---

## Publish Report

Publishes the report as the official event summary.

Once published:

* Viewer sees published version
* Report becomes read-only
* Report acts as historical record

---

## Regenerate Report

If tournament data changes after publication:

Example:

* Match reopened
* Score corrected
* Champion changes

System should display:

```text
Published report is out of date.

Please regenerate and publish the updated report.
```

Viewer continues to see previously published version until administrator republishes.

---

# Viewer Experience

Viewer only sees:

Published Event Report.

Viewer never sees:

* Preview
* Generate button
* Publish button
* Draft report

---

# Image Management

Images should NOT be stored inside report JSON.

Store images in object storage.

Example:

```text
event-reports/

    event2026/

        opening.jpg

        finals.jpg

        champions.jpg
```

The report stores only image references.

Example:

```json
{
    "photos":[
        "event2026/opening.jpg",
        "event2026/finals.jpg"
    ]
}
```

Benefits

* Smaller database
* Browser caching
* CDN support
* Reusable images
* Easy replacement

---

# Report Storage

Unlike Tournament State, the Event Report should be stored after publication.

Example key:

```text
btm_report_<eventId>
```

The report is a generated snapshot.

It is NOT the source of truth.

Tournament State remains the authoritative data.

---

# Regeneration Rules

Generate report when administrator requests.

Publish only after review.

If tournament changes after publication:

Report becomes stale.

Administrator must regenerate and republish.

Viewer always receives the latest published report.

---

# Performance Benefits

Instead of every viewer calculating statistics independently:

```text
100 Viewers

↓

100 Calculations

↓

Repeated Data Downloads
```

Use:

```text
Tournament State

↓

Generate Report Once

↓

Store Published Report

↓

100 Viewers Read Same Report
```

Benefits:

* Lower Supabase reads
* Faster loading
* Smaller network usage
* Consistent experience
* Better scalability

# Architecture

```text
Tournament State (Source of Truth)
                │
                ▼
      Generate Event Report
                │
                ▼
         Preview Report
                │
        Administrator Review
                │
                ▼
        Publish Event Report
                │
                ▼
Stored Report (Snapshot)
                │
                ▼
          Viewer Access
```

Tournament State continues to evolve during the event.

The Published Event Report is a generated, versioned snapshot intended for viewing, sharing, and preserving the event's legacy.

---

# Recommendation

Treat the Event Report as a publishable artifact rather than a dynamically generated page.

This provides:

* Better scalability
* Lower database load
* Faster viewer experience
* Consistent published content
* Support for rich media
* Historical preservation

The Tournament State remains the operational engine, while the Published Event Report becomes the official record and celebration of the event.
