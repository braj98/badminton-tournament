Biggest Remaining Design Problem
Event and Category Relationship

I found:

const cats = getCategories().filter(c => c.event === ev.name);

inside UI code.

This means:

Category still contains Event.

Example:

{
  "event": "BREN AVALON SPORTS MEET 2026"
}

This is the same architectural concern from previous reviews.

Why It Is A Problem

Current:

Category
  └── Event

Future preferred:

Event
  └── TemplateIds

Example:

Event

{
  "id":"sports_2026",
  "templateIds":[
      "junior_singles",
      "senior_boys"
  ]
}

Template

{
  "id":"junior_singles",
  "sport":"badminton"
}

Template should never know which event uses it.

Event should decide that.

Recommendation

Move towards:

Template
   ↓
Event
   ↓
Tournament State

instead of:

Template
   ↓
Category
   ↓
Event

Priority:

HIGH

Bugs / Risks
Bug 1

Event Rename Is Dangerous

Current:

renameEvent(oldName,newName)

updates:

ev.name

but categories appear linked by:

c.event === ev.name

Risk:

Renaming an event can orphan categories.

Example:

Before:

BREN AVALON SPORTS MEET 2026

After:

Summer Games 2026

Categories still contain:

BREN AVALON SPORTS MEET 2026

Result:

Categories disappear.

Priority:

HIGH

Bug 2

Event Name Used As Key

Current:

AppState.event = ev.name

Recommendation:

Store:

AppState.eventId

and resolve name from event object.

Never use display name as identifier.

Priority:

HIGH

Bug 3

Template ID Generation

Current:

createTemplateId(label)

generates IDs from labels.

Example:

Junior Singles

↓

junior_singles

Risk:

Renaming label later causes confusion.

Recommendation:

Store:

{
  "id":"junior_singles",
  "name":"Junior Singles"
}

ID never changes.

Name can change.

Priority:

MEDIUM

Bug 4

Cloud Sync Metadata Race

Current:

syncMetadataToCloud()

writes:

templates
events
categories

in parallel.

Potential issue:

Partial success.

Recommendation:

Add:

metadataVersion
updatedAt

for future recovery.

Priority:

LOW
