Problem Statement

Current navigation is competition-centric.

Example:

Senior Doubles
→ Fixtures
→ Results
→ Champion

This works for players.

However spectators typically want:

What is happening now?
What just finished?
What is next?
Who already won?

They should not have to open every competition.

Recommended Event Dashboard

Event Dashboard becomes the landing page.

Example:

Sports Meet 2026

🔥 Live

📖 Recent Results

📅 Upcoming

🏆 Champions

Supabase Load Analysis
Bad Design

Every viewer:

Every 5 seconds

loads:

All categories
All matches
All tournaments

With:

100 viewers

This becomes expensive.

Recommended Design

Load event data once.

Compute:

Live Matches
Recent Results
Upcoming Matches
Champions

in browser memory.

Even Better

Whenever admin updates:

Match Started
Match Completed
Champion Declared

save once.

Viewers receive update through Supabase realtime.

Result:

1 write
Many listeners

instead of:

100 viewers polling
Additional Optimization

Maintain:

event_dashboard

object.

Example:

{
  "live": [],
  "recentResults": [],
  "upcoming": [],
  "champions": []
}

Update only when tournament state changes.

Recommendation For Current Scale

Current scale:

2-3 admins
50-200 viewers
1 active event

Do NOT build a separate feed service.

Do NOT create additional Supabase tables.

Do NOT poll every few seconds.

Instead:

Continue storing tournament state normally.
Use match status:
UPCOMING
LIVE
COMPLETED
Build Event Dashboard from existing data.
Use Supabase realtime updates when state changes.
Show:
Live
Recent Results
Upcoming
Champions

This gives the best user experience with minimal additional Supabase load


My recommendation is not to build a separate "feed" table right now. Your existing tournament state already contains everything needed. Generate the dashboard from the existing state, and use Supabase realtime only when an admin changes something. That keeps both complexity and Supabase usage very low
