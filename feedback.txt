Requirement: Introduce Organization Layer (Future-Proof Architecture)
Objective

Prepare the Tournament Manager platform for future growth without increasing complexity for current users.

Current hierarchy:

Event
→ Sport
→ Competition
→ Tournament

Proposed future hierarchy:

Organization
→ Event
→ Sport
→ Competition
→ Tournament

The Organization layer will be introduced in the data model first and exposed in the UI only when needed.

Problem Statement

Currently:

BREN AVALON SPORTS MEET 2026

acts as both:

Organization
Event

These are actually different business entities.

Example:

Organization:
Bren Avalon

Event:
Sports Meet 2026

Combining both concepts makes future expansion difficult.

Business Goals

Support:

Multiple annual sports meets
Multiple tournaments per year
Multiple sports
Multiple communities/apartments/clubs (future)

Without redesigning the platform later.

Proposed Hierarchy
Level 1 - Organization

Represents:

Apartment Complex
Sports Club
Resident Welfare Association
Corporate Sports League

Examples:

Bren Avalon
Prestige Lakeside
XYZ Sports Club
Level 2 - Event

Represents a sports festival or tournament event.

Examples:

Sports Meet 2026
Sports Meet 2027
Independence Day Tournament
Winter Games 2027

Events belong to exactly one Organization.

Level 3 - Sport

Examples:

Badminton
Table Tennis
Chess
Carrom

Sports belong to an Event.

Level 4 - Competition

Examples:

Badminton:

Junior Singles
Junior Doubles
Senior Boys
Senior Girls

Table Tennis:

Open Singles
Open Doubles

Chess:

Open Category

Competitions belong to a Sport within an Event.

Level 5 - Tournament Instance

Contains runtime tournament data:

Players
Groups
Fixtures
Standings
Knockout
Champion
Example
Bren Avalon
│
├── Sports Meet 2026
│   │
│   ├── Badminton
│   │   ├── Junior Singles
│   │   ├── Junior Doubles
│   │   └── Senior Boys
│   │
│   ├── Table Tennis
│   │   └── Open Singles
│   │
│   └── Chess
│       └── Open
│
└── Sports Meet 2027
Scope For Current Release
UI Scope

No Organization UI.

Users continue to see:

Event
→ Sport
→ Competition
→ Tournament

No additional complexity.

Data Model Scope

Introduce:

organizationId

into Event.

Example:

{
  "id": "sports_meet_2026",
  "organizationId": "bren_avalon",
  "name": "Sports Meet 2026"
}

Default value:

{
  "organizationId": "default"
}

for all existing data.

Database Changes
organizations
{
  "id": "bren_avalon",
  "name": "Bren Avalon"
}
events
{
  "id": "sports_meet_2026",
  "organizationId": "bren_avalon",
  "name": "Sports Meet 2026",
  "templateIds": []
}
templates

No change required.

{
  "id": "junior_singles",
  "sport": "badminton"
}
tournament_state

No change required.

{
  "eventId": "sports_meet_2026",
  "templateId": "junior_singles",
  "data": {}
}
Migration Strategy
Phase 1 (Current)

Add:

organizationId

to Event model.

Default:

default

No UI changes.

Phase 2

Create Organization model.

Support:

Create Organization
Edit Organization

Still hidden from standard users.

Phase 3

Expose Organization selection.

Navigation:

Organization
→ Event
→ Sport
→ Competition

Non-Goals

For current release:

Do NOT implement:

Organization management screens
Organization permissions
Organization-level reporting
Multi-organization login

These are future enhancements.

Benefits
Immediate Benefits
Future-proof architecture
Cleaner business model
Easier event management
Long-Term Benefits

Supports:

Multiple apartment complexes
Multiple clubs
Multi-tenant deployments
Historical event tracking

without redesign.

Recommendation

Implement only the data model change now:

organizationId

Keep the current user experience unchanged:

Event
→ Sport
→ Competition
→ Tournament

This provides future scalability while keeping the application simple for apartment users.
